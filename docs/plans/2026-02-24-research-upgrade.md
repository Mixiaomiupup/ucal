## 问题诊断（更新后）

两个层面的问题，四个角色视角的交叉验证：

**层面 1：调研流程**
- 搜索词直接用用户输入，没有问题锐化（美食家）
- 搜索维度单一，只搜"议题"，不搜"情感/场景/身份"（美食家）
- 串行读取，慢（架构师）
- 搜一次就定方向，没有自我修正（调研者）
- 输出是表格填空，空洞（所有人）

**层面 2：调研→设计的转化**
- 调研挖到了数据/分布，但挖不到"那条让人鼻酸的评论"（用户）
- 调研结论是"65%选A"而非"有个人在这件事上态度激烈到不可理喻的原因是..."（用户）
- 设计出的题目场景不够具体、选项不够有人味、选项间张力不足（用户+美食家）

## 改进方案（5 个改动）

### 改动 1：搜索前问题锐化（新增步骤）

**来源**：美食家 + 调研者共同提出

用户说"调研火锅蘸料"，不要直接搜"火锅蘸料"。先做两件事：

**a. 问题拆解**：把原始关键词拆成 2-3 个更锐利的子问题
```
原始："调研火锅蘸料"
拆解：
  - "选蘸料的人在做一个关于什么的决定？"→ 搜索词：火锅蘸料 选择
  - "蘸料偏好跟什么身份/经历挂钩？"→ 搜索词：南方人 北方人 火锅
  - "蘸料这件事上什么情绪最强烈？"→ 搜索词：火锅蘸料 争议 吵
```

**b. 多维搜索策略**（美食家提出的四维模型）：

| 搜索维度 | 目标 | 搜索词构造方式 | 例子 |
|---------|------|--------------|------|
| 议题搜索 | 观点和争议 | 原始关键词 | "火锅蘸料" |
| 情感搜索 | 情感投射和共鸣 | 关键词 + 情绪词 | "火锅 想家"、"一个人吃火锅" |
| 场景搜索 | 行为模式 | 关键词 + 具体场景 | "第一次请对象吃火锅" |
| 身份搜索 | 身份认同 | 关键词 + 身份标签 | "北方人 在南方 火锅" |

**实操**：不需要四个维度都搜。问题锐化时选择 1-2 个最相关的维度，生成 1-2 个语义变体搜索词。第一轮搜索用原始词 + 1 个变体词，合并去重后选帖。

**涉及文件**：`~/.claude/skills/ucal/SKILL.md` — 在 Research 流程 Step 1 之前新增 Step 0

### 改动 3：两阶段反思（取代变长循环）

**来源**：架构师提出"固定两阶段"，调研者提出"证据追踪表"，两者合并。

**架构师的核心论点**：0-2 轮反思循环是一个 LLM 无法可靠执行的状态机。最大风险是"eager pursuit"——LLM 永远能找到"惊喜"，导致无限追搜。

**解法**：固定两阶段，SUFFICIENT 是默认值。

```
Phase 1（必做）：
  Step 0: 问题锐化 + 多维搜索策略
  Step 1: 搜索（原始词 + 变体词，合并去重）
  Step 2: 选帖（用选帖标准矩阵，展示给用户确认）
  Step 3: Pre-read hypothesis（读之前写 2-3 句预期）
  Step 4: 深读 3-5 篇（按平台策略串行/并行）
         每篇读完立即输出 3-5 行 interim summary
  Step 5: 证据追踪表 + 反思决策

Phase 2（可选，高门槛触发）：
  仅当 Phase 1 反思明确指出一个具体缺口 + 具体搜索词时触发
  追搜 2-3 篇 → 更新证据追踪表 → 直接进入综合
  不再做第二轮反思。
```

**Step 3: Pre-read hypothesis**（架构师建议，在读帖之前写）：

> **Pre-read hypothesis**: Based on the search results, I expect to find [具体预期]. The main perspectives are likely [X] and [Y]. I'm uncertain about [Z].

必须在调用 `ucal_platform_read` 之前写出。这样 Phase 1 反思才有一个真实的锚点。

**Step 5: 证据追踪表 + 反思**（调研者建议，替代 before/after diff）：

```markdown
**证据追踪表**

| 发现/判断 | 来源证据（引用原文） | 支撑帖数 | 对立证据 | 信息缺口 |
|-----------|-------------------|---------|---------|---------|

**反思决策**
- Pre-read hypothesis 中被证实的：[列出]
- Pre-read hypothesis 中被推翻的：[列出，附具体证据]
- 意外发现（hypothesis 中没有预料到的）：[列出，附具体证据]
- 决策：SUFFICIENT / PURSUE "[具体搜索词]" because [一句话理由]
- 已读帖子 URL 列表（计数：N 篇）

规则：
- 如果"被推翻的"和"意外发现"均为空 → 决策必须是 SUFFICIENT
- SUFFICIENT 是默认值。只有能说出具体缺口和具体搜索词时才选 PURSUE
- 如果已读 ≥6 篇 → 决策必须是 SUFFICIENT
```

**选帖标准矩阵**（调研者建议，取代模糊的"覆盖度"）：

```
从搜索结果中选帖，每个维度至少覆盖一个：
1. 互动量梯度：至少 1 篇高互动 + 1 篇低互动
2. 立场分布：如可判断，至少 1 篇与主流不同
3. 作者多样性：避免同一作者多篇
展示给用户时标注每篇入选理由。
```

**反思信号**（精简为 3 条二值判断 + 1 条证据质量）：

| # | 信号（是/否） | 行动 |
|---|-------------|------|
| 1 | 所有帖子观点方向一致？ | PURSUE：搜反面关键词 |
| 2 | 出现 ≥2 个未预期的专有术语/标签？ | PURSUE：用新术语搜 |
| 3 | 多数观点无具体经验/故事/案例支撑？ | PURSUE：搜"经验分享""测评""真实经历"类帖子 |
| 4 | 以上均"否"？ | SUFFICIENT |

**总阅读量上限：6 篇**（架构师建议：第 7-8 篇的边际价值低于上下文污染成本）。

**涉及文件**：`~/.claude/skills/ucal/SKILL.md` — 重写 Research 模式全部流程

### 改动 4：叙事输出 + 结构性诚实约束

**通用叙事框架**（跨平台共用）：

```markdown
# [话题] 调研报告
> [平台] | N 篇内容（基于有限样本） | YYYY-MM-DD

## 一句话结论
[最核心的发现，带态度]

## 我发现了什么
[叙事体。按洞察惊喜度排列，不按帖子顺序。]

每个发现用以下结构：
> [直接发现] / [推断]：[判断内容]（N/M 篇帖子，K 条评论提及）
> 证据："[引用原文/评论]" —— @用户X（帖A）
> 反例信号：如果发现 [具体反例]，则此结论需修正为 [修正方向]

如果没有意外发现，直接写"这个话题的社区讨论相当一致，无出乎意料的观点"。

## 扎心原文（新增节）
[从所有评论中摘出 3-5 条最有情感冲击力的原始评论，
包括让人鼻酸的、让人爆笑的、让人想吵架的。
这些是最有可能直接转化为下游设计素材的原料。
标注出处和互动量。]

## 社区真正在争什么（有争议时写，无争议则省略）
[挖掘争议背后的深层原因。
不是"正方 vs 反方"，而是"分裂的断层线是什么"。]

## 如果继续挖
[2-3 个方向 + 理由]

## 反思过程
[Pre-read hypothesis → 搜了什么 → 读了什么 → 证据追踪表 → 追搜了什么/为什么没追搜]

## 数据来源
| 标题 | 作者 | 链接 | 互动量 | 入选理由 |
```

**平台差异化**（在"我发现了什么"节内嵌不同子结构）：

| 平台 | 子结构 |
|------|--------|
| XHS | 按"讨论线程"组织：追踪评论区对话链的完整脉络（父评论→子评论的观点转折） |
| 知乎 | 按"论证比较"组织：答主A vs 答主B的论据类型和说服力对比 |
| X | 按"观点+传播"组织：原始观点 + 转赞比异常分析 |

**结构性诚实约束**（调研者建议，替代语义承诺）：

| 原来的语义约束 | 改为结构性格式约束 |
|-------------|----------------|
| "不夸大样本量" | 每个判断后强制标注 `（N/M 篇帖子，K 条评论提及）` |
| "标注不确定" | 区分两种标记：`[直接发现]` = 原文明确表达；`[推断]` = 交叉分析结论 |
| "证据优先" | "判断 + 引用"交替结构，无引用的判断不允许出现 |
| "不填空节" | 保留，可选节标注"如果..."省略条件 |

**涉及文件**：`~/.claude/skills/ucal/SKILL.md` — 重写所有 Research 输出模板

### 改动 5：搜"扎心原文"的意识（美食家+用户共识）

这不是一个独立的技术改动，而是贯穿整个流程的意识注入。

**核心洞察**（用户原话）：
> "能让我'哇'的调研发现不是'多少人选了什么'，而是'为什么有人在这件事上态度激烈到不可理喻'——是那些评论区里吵起来的瞬间、那些背后跟成长记忆和亲密关系挂钩的个人故事。"

**在 skill 中的体现**：
- Step 4 深读时，每篇帖子的 interim summary 必须包含：该帖/评论区中最具情感冲击力的 1 条原文引用
- 综合分析时，新增"扎心原文"节，专门汇集这类素材
- 这些原文是调研报告最有价值的部分——它们可以直接成为下游设计的场景描述和选项措辞

**在搜索策略中的体现**：
- 多维搜索中的"情感搜索"维度优先级提高
- 选帖时，评论区出现强烈情绪反应（大量回复、争吵、共鸣）的帖子优先入选

**涉及文件**：`~/.claude/skills/ucal/SKILL.md` — 分散在多个环节中

## 验证方式

1. **功能验证**：`调研一下"年轻人第一次做饭"`（XHS）
   - [ ] 有问题锐化步骤（拆出子问题 + 变体搜索词）
   - [ ] 有选帖标准（每篇标注入选理由）
   - [ ] 用户可以确认/调整选帖
   - [ ] 读帖前有 pre-read hypothesis
   - [ ] 每篇读完有 interim summary（含 1 条扎心原文）
   - [ ] 有证据追踪表
   - [ ] 反思决策有明确 SUFFICIENT/PURSUE 及理由
   - [ ] 输出是叙事式，有"扎心原文"节
   - [ ] 判断均有证据量级标注
   - [ ] 可选节如无内容则省略

2. **质量对比**：用 shige 原始话题 `火锅蘸料` 重新调研
   - 判断标准：报告中是否出现了 v1 调研没挖到的新维度
   - 判断标准："扎心原文"节的评论是否能直接启发题目设计

3. **边界测试**：测试一个无争议话题
   - 检查：是否诚实省略"社区真正在争什么"节

4. **反思可靠性测试**：
   - 检查：pre-read hypothesis 是否在读帖之前写出
   - 检查：证据追踪表是否每项有具体引用
   - 检查：如果追踪表中"被推翻"和"意外发现"均空，是否自动 SUFFICIENT
